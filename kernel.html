The Kernel: AI governance proved, not tested

The AxoDen Kernel is the control layer around an AI system. It limits how uncertainty builds up, checks that safety components are genuinely independent, and produces evidence that an assessor can review. It does not try to prove a model is perfect. It proves that the surrounding governance system stays within defined bounds.

Runtime posture tracked in canon 10 theorems · 6 formally proved DO-178C · IEC 61508 · DEF STAN 00-56 DOI: 10.5281/zenodo.18633407

In plain English

The Kernel decides what evidence must exist before an AI system is trusted, what limits it must stay inside, and what must be logged for later review.

What the Kernel does in practice

It proves that backup safety components are not failing in the same way

A safety architecture is not meaningfully redundant if every checker shares the same weakness. The Kernel treats independence as a design requirement, not a hope. The CMRI bound is the formal rule used to express that requirement for technical reviewers.

CMRI := supω∈Ω maxi<j ρ̄ij(ω) ≤ 1 − ISmin + εA

In plain terms, this asks a simple question: if one safeguard fails, how likely is it that the others fail for the same reason? ISmin is the minimum independence level required for a deployment profile, and εA is the small design tolerance allowed by the architecture. The Kernel is built to engineer and verify this bound, not measure it after the fact and hope for the best.

Three controls the Kernel enforces

Local checks that add up to a system-level safety case

The Kernel is built around three controls. Each one can be reviewed on its own, and together they form the broader safety case. When one component changes, only the local control around that component needs to be checked again.

EBDP

Entropy-Bounded Data Pipelines

Limits how uncertainty grows

Each pipeline stage has a defined ceiling for uncertainty. If a stage goes beyond that ceiling, the system treats it as a clear control breach rather than a vague warning sign. Applied to sensor fusion, RAG pipelines, multi-step reasoning, and RF ingestion. DOI: 10.5281/zenodo.17709557

EFI

Entropy-Frugal Intelligence

Limits unsupported model influence

A model is only allowed to add a bounded amount of information that is not already supported by the evidence it received. That means hallucinations are constrained by design, not merely watched after they happen.

CFS

Coprime-Factor Security

Builds independence into trust layers

The Kernel separates trust across cryptographic roots, vendors, operations, and hardware so one shared weakness cannot silently invalidate the whole safety case. DOI: 10.5281/zenodo.17690358

Runtime enforcement

Three runtime control modules

The Kernel uses three runtime modules to enforce its controls. Together they decide what state the system is in, what actions are allowed, and what evidence must be kept.

ARV

Adaptive Runtime Verifier

Autonomous Recovery Valve

A fail-closed policy engine. When several rules apply, it follows a clear priority order so the system does not drift into ambiguous behaviour under pressure.

MQ

Monitoring Quad

Four explicit operating states — no implicit state

A simple four-state operating model. Every governed component is always in one known state, and every state change is logged as a Canonical Attestation Record.

VSR

Verification & Safety Register

Certificate-based topology verification

Keeps the current topology certificate for the evidence graph and lets an independent reviewer verify it without recreating the whole graph construction process.

From deployment to certification

The certification pathway

These are the steps from initial deployment to certification-grade evidence submission.

01

Select ASIL-M conformance profile

Choose L1 through L4 based on consequence severity and applicable standards. The profile determines which artefact categories are required and the ISmin threshold the design must satisfy.

Deliverable: conformance profile selection document

02

Configure EBDP and EFI bounds

Define Hmaxk and εk for each pipeline stage based on domain physics and information flow requirements. The Kernel enforces these bounds at runtime and logs every violation as a CAR.

Deliverable: entropy budget document; stage-level EFI ceiling specification

03

Commission CFS independence analysis

Document coprime monitor root selection across the four independence planes. Produce ISatt measurement demonstrating compliance with ISmin at the chosen ASIL-M level.

Deliverable: CFS independence report; ER-cfs audit artefacts; CMRI bound verification

04

Generate and verify ER-topo.cert certificate

Construct the rotation system certificate for the evidence graph topology. Any independent party can verify it in O(E) time. This is the structural safety proof — the certificate that distinguishes AxoDen from every preceding framework.

Deliverable: ER-topo.cert certificate file; verifier test results; topological slack S measurement

05

Commission TAPL policy and produce CAR baseline

Formalise governing policies in TAPL. Run the system under the target workload profile and produce the baseline CAR chain demonstrating deterministic behaviour. Bit-exact replay verification confirms determinism across runs and platforms.

Deliverable: TAPL policy corpus; CAR chain baseline; replay verification log

06

Submit artefact bundle to assessor

The assembled artefact bundle maps directly onto evidence categories existing safety assessors already evaluate under DO-178C, IEC 61508, or DEF STAN 00-56. No new certification path. No novel evidence category. The assessor already knows what to do with this.

Deliverable: complete ASIL-M artefact bundle; assessor-ready evidence package
