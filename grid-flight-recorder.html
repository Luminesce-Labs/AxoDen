It enforces one contract: no irreversible physical actuation occurs unless safety, policy, and evidence conditions are all valid at the moment of execution. It is not a passive audit log. It is an active enforcement boundary.

The structural failure class that three frontier AI systems missed

The Zombie Intent Hazard is not a cybersecurity vulnerability or an edge case. It is a fundamental structural failure class in any system where AI decisions are queued, buffered, or persisted before physical execution. It applies to power grids, autonomous vehicles, medical devices, industrial control systems, autonomous drones, and any system with write-ahead intent and physical actuation.

The formal state desynchronisation

TLA+ formal verification

Three safety invariants machine-verified across 2.4 million reachable states

The specification covers the complete intent lifecycle: PROPOSED → VERIFIED → AUTHORISED → ACTUATED / BLOCKED → EXPIRED. Zero counterexamples across all reachable states for all three invariants.

Invariant 1 — Anti-Zombie

Freshness Exclusion

□ ∀ i ∈ Intents :
  state(i) = AUTHORISED
  ⇒ session(i) = CurrentBootSession

No intent may reach the AUTHORISED state unless its session binding matches the current live boot-session ID. The Zombie Intent Hazard is excluded by construction. This invariant holds across all 2.4 million verifiable states.

Invariant 2 — Quorum Gate

Quorum Integrity

□ ∀ i ∈ Intents :
  state(i) = AUTHORISED
  ⇒ |FreshAttestations(i)| ≥ ⌈(N+1)/2⌉

No intent may be authorised unless a strict sensor quorum Q has attested freshness of the current context. Single-sensor dropout or drift cannot unilaterally satisfy the authorisation condition. Risk-Adaptive Quorum Thresholds (RAQT, D84) extend this with deterministic adaptive thresholds.

Invariant 3 — Audit Completeness

Gapless CAR Ledger

□ ∀ i ∈ Intents :
  (state(i) = PROPOSED
  ∧ state'(i) ≠ PROPOSED)
  ⇒ |CARs(i)| = |CARs(i)|_prev + 1

Every transition out of the PROPOSED state produces exactly one Canonical Attestation Record. The audit ledger is gapless by construction. There is no execution path that advances intent state without a corresponding CAR entry.

Core mechanisms

Four enforcement components that compose into a complete actuation firewall

EOA-Latch

Evidence-or-Actuate Latch

The single most important architectural constraint in the Grid Flight Recorder: no physical actuation occurs unless AxoDen mathematically verifies context freshness, safety invariants, and sensor quorum integrity at the moment of execution. The EOA-Latch is not a check before actuation. It is the actuation precondition. Evidence existence is logically prior to physical action.

EOA-Latch state machine:
PENDING → EVIDENCE_CHECK → QUORUM_CHECK → FRESHNESS_CHECK
  → AUTHORISED → ACTUATED (signed auth token emitted)
  → BLOCKED → DENY_CAR (written, intent invalidated)

Reboot-Safe Session Binding

Volatile Boot-Session ID Invalidation

Queued intents are strictly bound to volatile boot-session IDs. A reboot, power cycle, or controlled restart mathematically invalidates all previously queued intents by invalidating the session ID. Commands computed under old state cannot execute under new state. This is the direct structural countermeasure to the Zombie Intent Hazard.

boot_session_ID := HMAC-SHA256(device_key, boot_timestamp || entropy_seed)
All queued intents carry session_ID binding
Mismatch: DENY + write DENY_CAR + force re-evaluation

OTT — D81

One-Time Actuation Tickets

Class H (high-consequence) actions are authorised by single-use tokens. Once consumed, the token is spent. An adversary cannot replay a captured authorisation to trigger a second actuation. The token is bound to the specific intent, the current session, and the current quorum attestation.

Cryptographically scoped single-use token binding intent identity, boot session, and quorum attestation
Consumption index recorded in CAR chain
Replay attempt: DENY + forensic alert

ACFF — D83

Attested Control-Flow Fingerprint

Firmware hashes prove code presence, not execution path. The ACFF binds the policy digest, the WASM module hash, the configuration hash, and the precedence traversal indices into a single verifiable fingerprint. An auditor can verify not only what code ran but how it decided.

Verifiable fingerprint binding the policy digest, code module, configuration, and decision path traversal into a single auditable record
Bound into CAR at every authorisation
Verifiable by any party with the ACFF and the CAR
